进入 **Week 3**，我们正式开始攻坚 **《工具拦截器与策略路由实现（Tool I/O Validation & Dynamic Mediation）》**。

在大规模工业级 AI 安全中，仅做 I/O 文本过滤是不够的。当 AI 演进到 Agent 阶段，最大的安全威胁来自于模型在调用外部工具（Tools）时，由于遭到**提示词注入（Prompt Injection）**或发生**目标漂移（Goal Drift）**，从而执行了不可逆的、具备高风险副作用的动作（如恶意删库、未授权外发资金、篡改系统配置）。

本周的战略目标是：**在底层构建一套零拷贝、不可旁路的 `ToolInterceptor`，并实现从 `Allow` 到 `AskConfirmation` / `Redact` 的风险调解策略路由（Dynamic Mediation Algebra）**。

## 一、 Week 3 敏捷开发计划 (Sprint Plan)

### 3.1 核心任务拆解

* **Task 1: 工具输入/输出校验器（Tool I/O Interceptor）开发**
* 捕获 Agent 发出的工具调用意图、参数 Schema，以及工具执行后返回给模型的 Payload。

* **Task 2: 副作用控制闸门（Side-Effect Gate）机制实现**
* 建立动态拦截规则：若工具声明包含不可逆副作用（`SideEffect: true`），控制面必须强行将决策路由至人工确认（`AskConfirmation`）或安全阻断。

* **Task 3: 动态调解策略路由（Mediation Router）升级**
* 升级 Week 1 的评估内核，引入更高级的策略代数：根据风险置信度、动作危害度等多维向量，动态选择最优缓解动作（如 PII 数据做 `RedactThenAllow`，命令注入做 `Block`）。
  
  

### 3.2 时延性能指标（Latency SLO）

* **工具上下文构建 + 动态路由判定：** $\le 1.5\text{ms}$

## 二、 核心技术实现 (Technical Implementation)

我们在 `tianmu/toolgate` 目录下构建工具网关，并在 `tianmu/core` 下升级我们的动态决策路由。

### 2.1 工具拦截器体系：`tianmu/toolgate/interceptor.go`

这个模块充当大模型与外部系统连接的 **Reference Monitor（引用监视器）**。任何工具的调用参数都必须在此进行 Schema 强校验和意图压缩。

```go
package toolgate

import (
  "encoding/json"
  "errors"
  "fmt"
  "tianmu/core"
)

// ToolDefinition 工业级工具注册声明
type ToolDefinition struct {
  Name        string                 `json:"name"`
  Description string                 `json:"description"`
  HasSideEffect bool                 `json:"has_side_effect"` // 是否包含不可逆外部物理世界影响
  ParamSchema map[string]string      `json:"param_schema"`    // 参数强类型定义
}

type ToolInterceptor struct {
  registry map[string]ToolDefinition
  engine   *core.Engine
}

func NewToolInterceptor(eng *core.Engine) *ToolInterceptor {
  return &ToolInterceptor{
    registry: make(map[string]ToolDefinition),
    engine:   eng,
  }
}

// RegisterTool 注册受信任的工具边界
func (ti *ToolInterceptor) RegisterTool(tool ToolDefinition) {
  ti.registry[tool.Name] = tool
}

// InterceptCall 在工具真正执行前实施不可旁路的安全审计与决策路由
func (ti *ToolInterceptor) InterceptCall(
  sessionID string,
  toolName string,
  rawParams string,
  inboundSignals []core.DetectorSignal,
) (*core.ControlDecisionObject, error) {

  // 1. 安全边界检查：拒绝调用未注册的幽灵工具（防止 Tool Injection 伪造工具名）
  def, exists := ti.registry[toolName]
  if !exists {
    return nil, fmt.Errorf("unregistered_tool_execution_denied: %s", toolName)
  }

  // 2. 参数 Schema 基础合规性解析校验（防止注入参数引发解析器崩溃）
  var parsedParams map[string]interface{}
  if err := json.Unmarshal([]byte(rawParams), &parsedParams); err != nil {
    return nil, errors.New("tool_parameters_schema_malformed")
  }

  // 3. 构建大模型运行时的 Action 上下文
  reqCtx := core.RequestContext{
    ProductID:       "Qira",
    Language:        "zh-CN",
    InteractionType: "agent_loop",
  }
  idCtx := core.IdentityContext{ActorID: sessionID}

  // 标注数据流向：从大模型上下文流向外部应用 API
  dataCtx := core.DataContext{
    DataClassification: "personal_sensitive",
    ContainsPII:        false,
    Source:             "model_context",
    Destination:        "external_api",
  }

  // 映射工具 Action 特征
  actCtx := core.ActionContext{
    ActionType: "call_tool",
    ToolName:   toolName,
    SideEffect: def.HasSideEffect,
  }

  // 4. 路由至核心控制面引擎做代数决策收敛
  cdo, err := ti.engine.MediateInbound(reqCtx, idCtx, dataCtx, actCtx, inboundSignals)
  if err != nil {
    return nil, err
  }

  return cdo, nil
}

```

### 2.2 风险调解策略路由升级：`tianmu/core/evaluator_v2.go`

我们需要升级 Week 1 的 `evaluator.go` 核心判定逻辑，使其不仅仅具备“最高权重拦截”，还能支持根据风险类别和场景执行**多态调解路由（Risk-Adaptive Mediation Router）**。

我们通过为 `Evaluator` 增加一个覆盖机制，来实现更复杂的策略路由：

```go
package core

// EvaluateAdvanced 升级版高级多态策略路由
func (e *Evaluator) EvaluateAdvanced(risk RiskEvaluation, act ActionContext) (PolicyDecision, error) {
  // 1. 调用基础级联判定
  baseDecision, err := e.Evaluate(risk)
  if err != nil {
    return baseDecision, err
  }

  // 2. 策略路由代数扩展：如果是敏感信息泄露信号，且基础决策为 Allow，动态路由至 Redact 降级处理
  if baseDecision.Decision == Allow {
    for _, s := range risk.Signals {
      if s.Triggered && s.DetectorID == "chinese_pii" && s.Confidence >= 0.85 {
        return PolicyDecision{
          Decision:          RedactThenAllow,
          PolicyPackVersion: e.currentPack.Version,
          ReasonCode:        "cn_privacy_leakage_mediated_to_redact",
        }, nil
      }
    }
  }

  // 3. 强力熔断闸门：无论任何风险信号，只要动作声明了不可逆的副作用（如 SideEffect=true），决策一律升级为 AskConfirmation
  if act.SideEffect && baseDecision.Decision == Allow {
    return PolicyDecision{
      Decision:          AskConfirmation,
      PolicyPackVersion: e.currentPack.Version,
      ReasonCode:        "side_effect_action_requires_human_approval",
    }, nil
  }

  return baseDecision, nil
}

```

## 三   Week 3 自动化演练与集成验证

我们编写标准的工具集成测试用例，验证在遇到 **高风险副作用工具（如转账）** 以及 **敏感隐私泄露（PII）** 时，策略路由能否精准实现动态降级与熔断。

新建集成测试文件 `tianmu/toolgate/interceptor_test.go`：

```go
package toolgate

import (
  "testing"
  "tianmu/core"
)

func TestToolInterceptor_MediateMediationAlgebra(t *testing.T) {
  // 1. 初始化包含中文治理特性的 Policy Pack
  mockPack := core.PolicyPack{
    Version: "v1.3.0-tool-governance-pack",
    Rules: []core.PolicyRule{
      {
        RiskCategory:     "prompt_injection",
        ConfidenceThresh: 0.80,
        TargetDecision:   core.Block,
        ReasonCode:       "malicious_tool_injection_blocked",
      },
    },
  }
  eval := core.NewEvaluator(mockPack)
  engine := core.NewEngine(core.PersonalAI, eval)

  interceptor := NewToolInterceptor(engine)

  // 2. 注册一个具备高风险副作用的敏感工业级工具：企业网银转账
  transferTool := ToolDefinition{
    Name:          "enterprise_finance_transfer",
    Description:   "执行企业级资金转账账户划拨",
    HasSideEffect: true, // 声明副作用强闸门
    ParamSchema:   map[string]string{"amount": "float", "to_account": "string"},
  }
  interceptor.RegisterTool(transferTool)

  // 3. 场景 A 演练：大模型无明显恶意信号，但试图调度高副作用工具
  t.Run("Side_Effect_Gate_Trigger_Confirmation", func(t *testing.T) {
    cleanSignals := []core.DetectorSignal{} // 无明显外界安全报警信号
    params := `{"amount": 50000.0, "to_account": "622202******1102"}`

    cdo, err := interceptor.InterceptCall("session_agent_001", "enterprise_finance_transfer", params, cleanSignals)
    if err != nil {
      t.Fatalf("Interceptor deployment error: %v", err)
    }

    // 断言验证：策略路由必须将 Allow 强制熔断升级为 AskConfirmation
    if cdo.PolicyDecision.Decision != core.AskConfirmation {
      t.Errorf("Expected decision 'AskConfirmation' due to SideEffect Gate, got '%s'", cdo.PolicyDecision.Decision)
    }
    if cdo.PolicyDecision.ReasonCode != "side_effect_action_requires_approval" {
      t.Errorf("Mismatched reason code, got '%s'", cdo.PolicyDecision.ReasonCode)
    }
  })

  // 4. 场景 B 演练：恶意攻击者通过注入手段伪造未注册的隐藏工具试图越权执行
  t.Run("Deny_Unregistered_Ghost_Tool", func(t *testing.T) {
    params := `{"cmd": "rm -rf /"}`
    _, err := interceptor.InterceptCall("session_agent_001", "ghost_malicious_tool", params, nil)

    if err == nil {
      t.Errorf("Security Flaw: Engine should deny unregistered tool execution implicitly.")
    } else {
      t.Logf("[Pass] Successfully blocked ghost tool: %v", err)
    }
  })
}

```
